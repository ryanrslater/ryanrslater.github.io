Starting My Serverless Code Runner Project

Building a place where developers can upload code, let Kubernetes run it, and pay only for what they use.

Tags: Laravel Kubernetes Stripe Billing

This is the very first entry for my infrastructure journal, and it sets the tone for everything that comes next. I want to build an application where developers can upload zipped code, deploy it into a managed Kubernetes environment, monitor usage, and get billed seamlessly. Think of it as a lightweight serverless experience powered by the tools I love most.

What I’m Building

The goal is to deliver an interface that feels as easy as clicking “Upload” and “Deploy,” while hiding the complexity of containers, networking, and scaling behind the scenes. Users will start by creating an account, uploading a ZIP file containing their code, and choosing runtime defaults. From there, the platform will package the code, schedule it onto Kubernetes, expose an endpoint, and start tracking usage automatically.

Core Workflow

  • Account creation: Onboarding flows that feel friendly but enforce good security practices.
  • Code upload: Accept ZIP archives, validate contents, and store them securely for build and deploy steps.
  • Deploy: Kick off a container build, push it to a registry, and roll out a Kubernetes deployment with sensible defaults.
  • Monitor: Track invocation counts, execution time, and resource consumption with a simple dashboard.
  • Bill: Use Stripe to meter usage and invoice customers without manual intervention.

Stack Choices

I’m leaning on Laravel for the web application because of its expressive syntax and mature ecosystem for authentication, queues, and APIs. Kubernetes will orchestrate the workloads, giving me autoscaling and isolation without reinventing the wheel. Stripe will handle billing so I can focus on delivering a great developer experience instead of wrestling with invoices.

What Comes Next

Over the next few posts I’ll document the first user journey end-to-end: creating an account, uploading code, seeing it run, and getting usage metrics. Along the way I’ll capture the Kubernetes manifests, Laravel controllers, queue jobs, and billing hooks that make it all tick. This first entry is my stake in the ground—time to build.
